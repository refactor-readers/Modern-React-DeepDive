# 12.1 웹사이트와 성능
사용자는 목적 달성 가능성, 짧은 로딩 시간, 보안을 기대하며, 1초의 지연은 전환율에 약 4.42%의 손실을 입힌다.

# 12.2 핵심 웹 지표란?
구글이 조사한 사용자 경험의 필수 지표로, LCP, FID, CLS가 핵심

# 12.3 최대 콘텐츠풀 페인트 (LCP)
## 12.3.1 정의
페이지 로드 시작부터 뷰포트 내 가장 큰 이미지나 텍스트가 렌더링되는 시간을 의미

## 12.3.2 개선 방안
- 이미지 최적화: `<img>` 태그는 프리로드 스캐너에 의해 조기에 발견되므로 `background-image`보다 유리
- 서버 사이드 렌더링 : useEffect를 통한 데이터 로딩은 LCP를 늦추므로 서버에서 미리 빌드하는 것이 좋음

# 12.4 최초 입력 지연 (FID)
## 12.4.1 정의
사용자의 첫 상호작용(클릭 등)부터 브라우저가 이벤트 핸들러 처리를 시작하기까지의 대기 시간

## 12.4.2 기준 점수
FID는 사용자가 느끼는 '응답성'을 수치화한 것이므로, 다음과 같은 기준으로 점수를 판단
- Good(좋음) : 100ms 이내에 응답이 올 경우
- Needs Improvement (개선 필요) : 300ms 이내에 응답이 올 경우
- Poor (나쁨) : 300ms 초과할 경우

## 12.4.3 측정 도구
FID는 '실제 사용자'의 상호작용이 있어야만 측정 가능한 **필드지표(Field Metrics)**
따라서 일반적인 Lighthouse 점수보다는 실사용 데이터 기반의 도구가 중요
- Chrome User Experience Report (CrUX) : 실제 크롬 사용자들의 통계를 바탕으로 측정
- PageSpeed Insights: URL을 입력하면 실사용자 데이터(필드 데이터)와 실험실 데이터를 모두 보여줌
- **참고 : 실험실 도규(Lighthouse 등)에서는 FID 대신 **Total Blocking Time(TBT)*을 대리지표로 사용

## 12.4.4 나쁜 점수가 나오는 이유
FID가 높게(나쁘게) 나타나는 가장 주된 원인은 메인 스레드(Main Thread)의 차단
- 대규모 자바스크립트 실행 : 브라우저가 거대한 JS 번들을 파싱하고 실행하는 동안 사용자가 클릭을 하면, 브라우저는 실행 중인 작업을 마칠 때까지 이벤트 핸들러를 실행할 수 없음
- 긴 작업(Long Tasks) : 하나의 작업이 50ms 이상 소요될 경우 메인 스레드가 점유되어 사용자 입력에 즉각 반응하지 못함
- 타사 스크립트 (Third-party scripts) : 분석 도구나 광고 스크립트 등이 메인 스레드 우선순위를 가져갈 때 발생
