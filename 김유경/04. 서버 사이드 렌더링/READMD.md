# 4.1 서버 사이드 렌더링이란?

## 4.1.1 싱글 페이지 애플리케이션의 세상

서버 사이드 렌더링 애플리케이션과 반대되는 개념 = 싱글 페이지 애플리에이션

#### 싱글 페이지 애플리케이션(Single Page Application:SPA)이란?

렌더링과 라우팅에 필요한 대부분의 기능을 서버가 아닌 브라우저의 자바스크립트에 의존하는 방식을 의미

- 최초에 첫 페이지에서 데이터를 모두 불러온 이후에는 페이지 전환을 위한 모든 작업이 자바스크립트와 브라우저의 `history.pushState`와 `history.replaceState`로 이뤄지기 때문에 페이지를 불러온 이후에는 서버에서 HTML을 내려받지 않고 하나의 페이지에서 모든 작업을 처리함
  => 그래서 싱글 페이지 애플리케이션이라고 한다~

싱글 페이지 애플리케이션의 HTML 코드

- 사이트 접속하면 사이트가 나오지만 소스 보기로 HTML 코드를 봤을 때는 `<body />` 내부에 아무런 내용이 없다.

싱글 페이지 애플리케이션의 장단점

- 단점 : 최초에 로딩해야 할 자바스크립트의 리소스가 커짐
- 장점 : 최초 로딩 이후에는 서버를 거쳐 필요한 리소스를 받아올 일이 적어지기 때문에 사용자에게 훌륭한 UI/UX를 제공

#### 전통적인 방식의 애플리케이션과 싱글 페이지 애플리케이션의 작동 비교

과거 서버 사이드에서 작동하던 전통적인 방식

- 페이지 전환이 발생할 때마다 새롭게 페이지를 요청하고 HTML 페이지를 다운로드 해 파싱하는 작업을 거침
  => 페이지를 처음부터 새로 그려야해서 흰 화면이 깜박이는 현상이 발생

1. 전통적인 방식의 서버 사이드 애플리케이션 : 네이버

- 네이버 스포츠 화면은 네이버 홈과 다른 환경에서 HTML을 만들어서 제공하므로 어쩔 수없이 처음부터 HTML을 다시 완성해야 함

2. 완성도 높은 SPA 서비스 : Gmail

- 이메일 목록에서 이메일 메시지를 클릭하면 페이지 전환이 느껴지지 않을 정도로 매끄러운 화면 전환

#### 싱글 페이지 렌더링 방식의 유행과 JAM 스택의 등장

과거 PHP나 JSP(JavaServer Pages)를 기반으로 대부분 웹 애플리케이션이 만들어졌을 때는 거의 대부분의 렌더링이 서버 사이드에서 이뤄짐

- 페이지를 요청하면 서버에서 완성된 HTML을 내려받고, 또 페이지 이동이 있으면 새로운 페이지를 서버에서 내려받는 방식
  => 자바스크립트는 사용자에게 추가적인 경험을 주기 위한 보조적인 수단일 뿐

자바스크립트의 사용 범위가 확장되면서, 자바스크립트를 모듈화하는 방안이 논의되기 시작
=> `CommonJS`와 `AMD(Asynchronous Module Definition)`의 등장

이러한 자바스크립트의 모듈화, 사용자 기기 성능 향상과 인터넷 속도 발전으로 자바스크립트의 역할과 규모가 확장

**싱글페이지 애플리케이션이 널리 퍼지게 된 것은 단순히 사용자에게 좀 더 멋진 UX를 제공하는 것이 가능해져서만은 아니다.**

- PHP시절, 웹 애플리케이션을 만들기 위해서는 자바스크립트 외에도 신경 쓸 것이 많았지만 싱글 페이지 애플리케이션에서는 단지 브라우저 내부에서 작동하는 자바스크립트만 잘 작성하면 문제 없다.

- 프론트엔드 개발자들에게 좀 더 간편한 개발 경험을 제공
- 간편하게 웹 애플리케이션을 만들 수 있다.

**기본 웹 개발은 LAMP(Linux*운영체제, Apache*서버, MySQL*데이터베이스, PHP/Python 등* 웹 프레임워크) 로 구성**

- 서버 의존적인 처리는 기능 확장과 사용자 유입에 서버 확장을 필요로 했지만, 클라우드의 개념이 부족했던 당시에는 서버를 확장하는 것이 매우 번거로웠음

**JAM(JavaScript, API, Markup)의 등장**

- 프론트엔드는 자바스크립트와 마크업(HTML, CSS)을 미리 빌드해두고 정적으로 사용자에게 제공하면 이후 작동은 사용자의 클라이언트에서 실행되서 서버 확장성 문제에서 자유로워질 수 있었음
- 이와 함께 Node.js의 고도화로 `MEAN`(**M**ongoDB, **E**xpress.js, **A**ngularJS, **N**ode.js), `MERN`(MongoDB, Express.js, React, Node.js) 스택처럼 API 서버 자체도 자바스크립트로 구현하는 구조가 인기를 끌기 시작

#### 새로운 패러다임의 웹서비스를 향한 요구

사용자 기기와 브라우저 성능, 인터넷 속도는 향상되었지만, 서비스의 초기 로딩 속도는 동등하거나 그 이하이다.

- 향상된 만큼, 웹 서비스의 복잡성이 더 배로 높아지면서 더 많은 소스 코드가 초기에 로딩되고 있음

=> 제품의 웹 서비스 환경에 대해 한 번 더 고민해봐야 할 때

## 4.1.2 서버 사이드 렌더링이란?

최초에 사용자에게 보여줄 페이지를 서버에서 렌더링해 빠르게 사용자에게 화면을 제공하는 방식을 의미

- 여러 발전에도 불구하고 웹페이지가 점점 느려지는 상황에 대한 문제의식을 싱글 페이지 애플리케이션의 **태생적인 한계**에서 찾고, 이를 개선하고자 서버에서 페이지를 렌더링 해 제공하는 기존 방식이 다시금 떠오르고 있음

**싱글 페이지 애플리케이션과 서버 사이드 렌더링의 차이**
=> 💡 렌더링의 책임을 어디에 두느냐

1. 클라이언트 렌더링

- 사용자에게 제공되는 자바스크립트 번들에서 렌더링을 담당
- 사용자 기기의 성능에 영향을 받음

2. 서버 사이트 렌더링

- 렌더링에 필요한 작업을 모두 서버에서 수행
- 기기 성능에 비교적 덜 영향을 받으며 안정적인 렌더링이 가능

### 서버 사이드 렌더링의 장점

**1. 최초 페이지 진입이 비교적 빠르다**

- 화면 렌더링이 HTTP 요청에 의존적이거나 렌더링해야 할 HTML의 크기가 커진다면 상대적으로 서버 사이드 렌더링이 더 빠를 수 있음
- 물론! 서버가 사용자에게 렌더링을 제공할 수 있을 정도의 리소스가 확보되어 있어야 함

#### 검색 엔진과 SNS 공유 등 메타데이터 제공이 쉽다

👩🏻 이것은 가장 유명한 서버 사이드 렌더링의 장점이자 특징이지 않을까 싶습니다.
👩🏻 이 부분에 대해서 알아보려면 메타데이터를 수집하는 검색 엔진이 어떻게 데이터를 수집하는지 알아볼 필요가 있을 것 같습니다.

**검색 엔진이 크롤링하는 과정**

1. 검색 엔진 로봇(머신)이 페이지에 진입한다.
2. 페이지가 HTML 정보를 제공해 로봇이 이 HTML을 다운로드한다. 단, 다운로드만 하고 **자바스크립트 코드는 실행하지 않는다.**
3. 다운로드한 HTML 페이지 내부의 오픈 그래프(Open Graph)나 메타(meta) 태그 정보를 기반으로 페이지의 검색(공유) 정보를 가져오고 이를 바탕으로 검색 엔진에 저장한다.

클라이언트 렌더링은 자바스크립트를 실행해서 필요한 정보를 적용하기 때문에 자바스크립트를 실행하지 않는 검색 엔진 최적화에는 부적절하다.

#### 누적 레이아웃(CLS: Cumulative Layout Shift) 이동이 적다

화면이 보여진 뒤에 정보 추가로 인해 레이아웃이 덜컥 이동하는 부정적인 사용자 경험

- 클라이언트 렌더링은 페이지 콘텐츠가 API 요청에 의존하고, API 요청의 응답 속도가 제각각이기 때문에 이를 적절히 처리하지 않으면 CLS가 문제가 발생할 수 있음
- 서버 사이드 렌더링은 이러한 요청이 완전히 완료된 이후에 완성된 페이지를 제공하므로 이러한 문제에서 비교적 자유로움

#### 사용자의 디바이스 성능에 비교적 자유롭다

- 클라이언트 렌더링은 사용자의 디바이스에서 자바스크립트가 실행되기 때문에 이에 절대적으로 의존적

- 서버 사이드 렌더링을 수행하면 이러한 부담을 서버에 나눌 수 있어, 비교적 자유로워질 수 있음

그러나 **인터넷이 자체가 느리거나**, 사용자 방문 폭증으로 **서버가 느려지면** 느려질 수 있음

#### 보안에 좀 더 안전하다

JAN 스택을 채택한 프로젝트의 문제점은 보안
-> 애플리케이션의 모든 활동이 브라우저에 노출되는 것

- 클라이언트 렌더링은 개발자 도구로 API 호출과 인증 같은 작업이 노출될 수 있는 위험성이 있음 (별도의 작업 필요)
- 서버 사이드 렌더링은 인증 혹은 민감한 작업을 서버에서 수행하고 그 결과만 브라우저에 제공해 이러한 보안 위협을 피할 수 있음

### 서버 사이드 렌더링의 단점

#### 소스코드를 작성할 때 항상 서버를 고려해야 한다

소스코드 전반에 걸쳐 서버 환경에 대한 고려가 필요

- 전역 객체인 window 또는 브라우저에만 있는 전역 객체 sessionStorage이 문제
- 서버에서 실행될 가능성이 있는 코드에서는 window에 대한 접근을 최소화해야 한다.
  - ❗이 같은 처리는 라이브러리도 마찬가지라서, 라이브러리 사용에 사용 환경을 고려해야 함

#### 적절한 서버가 구축돼 있어야 한다

- 클라이언트 렌더링에서 서버는 HTML, 자바스크립트, CSS 리소스 등을 다운로드할 준비만 하면 됨
- 서버 사이드 렌더링에서 서버는 사용자의 요청을 받아 렌더링을 수행할 서버가 필요
  - 물리적인 가용량 확보 및 장애 상황에 대한 복구 전략
  - 요청 분산, 프로세스 다운에 대한 대비 (PM2와 같은 프로세스 매니저)

#### 서비스 지연에 따른 문제

- 클라이언트 렌더링에서는 느린 작업이 있어도 어떤 "화면이라도" 보여준 상태에서 느린 작업이 수행되기 때문에 `로딩 중`과 같은 안내로 사용자의 대기를 유도할 수 있음
- 서버 사이드 렌더링에서 최초 렌더링에 지연이 발생하면 아무것도 보여줄 수 없어 심각한 사용자 경험을 제공할 수 밖에 없음
  - 거즌 그림판 ㅋ

## 4.1.3 SPA와 SSR을 모두 알아야 하는 이유

### 서버 사이드 렌더링 역시 만능이 아니다

클라이언트에서 발생하는 모든 무거운 작업을 서버에 미루고, 작업이 모두 서버에서 이뤄진다고 해서 모든 성능 문제가 해결되는 것은 아님

잘못된 웹페이지 설계는 오히려 성능을 해치고, 관리 포인트만 늘릴 수 있음

### 싱글 페이지 애플리케이션과 서버 사이드 렌더링 애플리케이션

- Gmail과 같이 완성도가 매우 뛰어난 싱글 페이지 애플리케이션

  - 최초 페이지 진입 시에 **보여줘야 할 정보만 최적화**해 요청해서 렌더링
  - 이미자와 같은 중요성이 떨어지는 리소스는 **게으른 로딩**으로 렌더링에 방해되지 않도록 처리
  - **칼 같은 코드 분할**(code splitting 사용자에게 필요한 부분만 나눠서 번들링하는 기법)
    => 불필요한 자바스크립트 리소스의 다운로드와 실행을 방지
  - 라우팅이 발생하면 변경이 필요한 HTML 영역만 교체해 사용자의 피로감을 최소화
    => 멀티 페이지 애플리케이션 또한 마찬가지로 엄청난 최적화를 가미해도, 브라우저 API와 자바스크립트를 활용한 라우팅을 기반으로 한 매끄러운 라우팅보다 뛰어난 성능을 보여줄 수는 없을 것
    <br />

- 평균적인 싱글 페이지 애플리케이션은 평균적인 멀티 페이지 애플리케이션보다 느리다.
  - 멀티 페이지 애플리케이션은 매번 서버에 렌더링 요청을 하고, 서버는 안정적인 리소스를 기반으로 매 요청 마다 비슷한 성능의 렌더링을 수행할 것
    - 싱글 페이지 애플리케이션은 렌더링과 라우팅 최적화가 되어 있지 않으면 기기에 따라 성능이 들쑥날쑥할 것
    - 렌더링과 라우팅 최적화는 매우 어려운 작업

#### 결론

💡 가장 뛰어난 싱글 페이지 애플리케이션은 가장 뛰어난 멀티 페이지 애플리케이션보다 낫다.

**멀티 페이지 애플리케이션에서 발생하는 라우팅 문제를 해결하기 위한 API가 브라우저에 추가되고 있음**

- `back forward cache(bfcache)` : 같은 출처(origin)에서 라우팅이 일어날 경우 화면을 잠깐 하얗게 띄우는 대신 이전 페이지의 모습을 잠깐 보여주는 기법
  <a src="https://www.youtube.com/watch?v=_me7_7C6Drs">적용 예시</a>
- `Shared Element Transitions` : 페이지 라우팅이 일어났을 때 두 페이지에 동일 요소가 있다면 해당 콘텍스트를 유지해 부드럽게 전환되게 하는 기법
  <a src="https://developer.android.com/develop/ui/compose/animation/shared-elements?hl=ko">적용 예시 </a>

이러한 기법은 모두 싱글 페이지 애플리케이션에서 구현 가능하지만, 완벽하게 구현하려면 자바스크립트와 CSS 등의 도움을 받아야하고, 상당한 리소스가 소모된다.
=> 평균적인 노력으로 평균적인 사용자 경험을 제공한다고 가정하면 별도의 최적화를 거쳐야 하는 싱글 페이지 애플리케이션보다 서버에서 렌더링되는 멀티 페이지 애플리케이션이 더 나은 경험을 제공

### 현대의 서버 사이드 렌더링

`LAMP`의 서버 사이드 렌더링과는 다름

- `LAMP` 스택

  - 장점 : 모든 페이지 빌드를 서버에서 렌더링해 초기 페이지 진입이 빠름
  - 단점 : 라우팅이 발생할 때도 마찬가지고 서버에 의존해야 하기 때문에 싱글 페이지 렌더링 방식에 비해 라우팅이 느림

- 현대의 서버 사이드 렌더링
  > 이 두가지 장점을 모두 취한 방식으로 작동
  - 최초 웹 사이트 진입 시에는 서버 사이드 방식으로 서버에서 완성된 HTML을 제공 받음
  - 이후 라우팅에서는 서버에서 내려받은 자바스크립트를 바탕으로 마치 싱글 페이지 애플리케이션처럼 작동
    - Next.js, Remix 등이 이러한 방식으로 동작
